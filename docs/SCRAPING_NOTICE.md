# 爬虫法律与合规注意事项

## TL;DR

**不要直接对大众点评 / 小红书 / 抖音运行 v1 仓库里的爬虫代码。** 这些站点的 ToS 明确禁止自动化采集,且都部署了主动反爬。继续之前请走以下决策树:

```
你需要 reviews 数据吗?
├─ 是 → 优先级:
│   1. 公开数据集(如有,见 §下游资源)
│   2. 微博公开 API + 学术研究授权
│   3. 自建小规模数据 (n < 100, 手工记录, 不算"自动化爬取")
│   4. 与品牌/平台正式合作获取数据
│   5. (如以上都不行) 谨慎评估 ToS 后再考虑爬虫
└─ 否 → MockScraper / LocalFileScraper 足够开发用
```

## 已知风险

| 平台 | ToS 状态 | 反爬强度 | 备注 |
|---|---|---|---|
| 大众点评 | 明确禁止自动化采集 | 高 (动态选择器 + IP 封禁 + 验证码) | 高频访问可能被封 IP,严重者品牌账号封禁 |
| 小红书 | 明确禁止 | 极高 (设备指纹 + 行为检测 + 笔记限流) | 笔记/评论排序对外部访问不稳定 |
| 抖音 | 明确禁止 | 高 (设备指纹) | 视频评论需要登录态,数据量大 |
| 微博 | 公开 API 可用 | 低 | 有正式 API,但需要开发者审批;评论质量比茶饮内容稀疏 |

## 中国法律相关

- **《数据安全法》** (2021):跨境/敏感数据传输需审批
- **《个人信息保护法》** (2021):任何包含可识别用户身份的字段(用户名/头像/UID)都需要合法基础;研究用途的"合理使用"边界在中国法律下并不明确
- **《反不正当竞争法》**:大规模数据采集可能被认定为"以技术手段擅自使用他人合法成果"
- **学术研究豁免**没有明确成文条款。如果产出涉及商业化,法律风险显著上升

## 如果你仍然要采集

1. **明确认识风险**:个人 / 团队需自行承担法律后果
2. **数据脱敏**:任何用户标识(用户名/头像/uid)在落库前用 `hash_user_id()` 单向哈希
3. **限速**:adapters 默认 `rate_limit_sec ≥ 3` 秒;不要降低
4. **不要发布原始数据**:仓库 `.gitignore` 已排除 `data/reviews/*.parquet`,保持现状
5. **论文/报告中匿名化数据来源**:不要写"我们爬取了大众点评的 X 条评论",写"我们使用了来自公开评论平台的 X 条匿名化文本"
6. **保留原始 ToS 抗辩证据**:截图/存档每次 ToS 变更,如有诉讼可证明合理使用意图

## 替代方案(推荐)

| 方案 | 量级 | 法律风险 | 数据质量 |
|---|---|---|---|
| **手工记录** (个人/室友/小组) | 100-500 条 | 0 | 高 (有明确同意) |
| **公开数据集** (HuggingFace / Kaggle / 学术论文) | 5K-50K 条 | 低 | 中 (主题可能不完全匹配) |
| **微博公开 API** | 视配额 | 低 | 低-中 (奶茶讨论稀疏) |
| **与品牌合作** | 视协议 | 0 | 高 (但有商业绑定) |
| **MockScraper + 合成数据** | 任意 | 0 | 低 (用于开发/Stage 1 预训练原型,真实训练前需替换) |

## 紧急停止

如果运行中收到平台的合规通知:

1. **立即停止**爬取脚本
2. **从 git history 中清除**已采集的数据(`git filter-repo`)
3. **不要重启**直到完成 ToS 评估
4. 如适用,**回复平台说明用途**

---

**底线**:本项目代码默认假设你已审慎处理上述合规问题。`MockScraper` 和 `LocalFileScraper` 是无风险的,任何使用 `DianpingScraper` / `XiaohongshuScraper` 的责任完全在用户。
